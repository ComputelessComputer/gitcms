# AGENTS.md — gitcms contributor guide for AI agents

You're an AI coding agent (Claude Code, Codex, Cursor, Copilot, etc.) working on the gitcms codebase. Read this file first. It's the source of truth for how this repo is built, what's already opinionated, and what landmines to avoid.

> **Looking to use gitcms as a CMS in another project?** That's a different doc — see [`docs/AGENTS-USING-GITCMS.md`](docs/AGENTS-USING-GITCMS.md).

## What gitcms is

A self-hostable, AI-friendly CMS where:

- **Content** lives in a GitHub repo as plain markdown (the "content repo"). Content edits become commits.
- **Media** lives in a pluggable storage backend (Supabase / S3 / R2 / GitHub itself / local disk).
- **Auth** is decoupled: an `AuthAdapter` decides who's signed in, a `MembersProvider` decides who's allowed, a `GitHubTokenSource` decides which token writes content commits. These three are orthogonal — you can pair them however you like.

The whole project is intentionally small, vendor-neutral, and config-driven. Resist the urge to add abstractions or new vendors without a real user pulling.

## The three primitives

When in doubt, ask which of these you're touching. Don't conflate them.

| Primitive           | Lives in       | Purpose                                                                  |
| ------------------- | -------------- | ------------------------------------------------------------------------ |
| `AuthAdapter`       | `src/auth/`    | Proves who is signed in. GitHub OAuth or JWT today.                      |
| `MembersProvider`   | `src/auth/`    | Decides which signed-in identities are allowed in. Env-driven today.     |
| `GitHubTokenSource` | `src/auth/`    | Returns the GitHub token used for content commits. OAuth or service PAT. |
| `StorageAdapter`    | `src/storage/` | Reads/writes media blobs. Supabase, S3, GitHub, local.                   |

These are interfaces. New adapters slot in by implementing the interface and registering in the factory. Don't fork the consumer — extend the interface.

## Stack

- **Framework**: TanStack Start (Vite + React 19, server functions)
- **Routing**: TanStack Router (file-based, with `routeTree.gen.ts` regenerated by `pnpm routegen`)
- **Editor**: Tiptap (markdown serializer in `src/client/tiptap/`)
- **Validation**: zod everywhere
- **Tests**: vitest
- **Lint**: eslint with custom rules — see Landmines
- **Format**: prettier
- **Package manager**: pnpm 10.33 — do NOT use npm or yarn
- **Node**: 20+

## Common workflows

### Setup

```bash
pnpm install
cp .env.example .env                      # fill in values
cp examples/gitcms.config.example.ts gitcms.config.ts
```

### Dev loop

```bash
pnpm dev                                  # vite dev server on :3000
pnpm typecheck                            # tsr generate + tsc --noEmit
pnpm test                                 # vitest run
pnpm lint                                 # eslint .
pnpm format                               # prettier --check (use --write to fix)
pnpm build                                # production build
```

### Before pushing

```bash
pnpm typecheck && pnpm test && pnpm lint && pnpm format && pnpm build
```

CI runs all five in a parallel matrix. Don't push red.

## Conventions

### Commits

Use [Conventional Commits](https://www.conventionalcommits.org/): `feat:`, `fix:`, `chore:`, `docs:`, `ci:`, `refactor:`, `test:`. Scope is optional but encouraged: `feat(auth): add ...`. PRs squash-merge into `main`.

If you're an agent committing on behalf of a user, include a co-authored trailer:

```
Co-authored-by: Real Person <real@example.com>
```

### PR descriptions

Short. Explain the _why_, not the _what_. Top section: a one-paragraph summary. Then sections only if needed. Avoid status-report formatting (no "Stand-up notes"). Don't paste tool output into the PR — link to the run.

When opening PRs from the CLI, **always** use `gh pr create --body-file <path>` instead of `--body "..."` — backticks and code fences in inline strings get mangled by the shell before `gh` sees them.

### Server functions vs HTTP routes

- Use `createServerFn` (in `src/server/fn/`) for typed, RPC-style endpoints called from the React app.
- Use raw HTTP routes (in `src/routes/api/`) for blob transfer (media upload/download) where you need full request/response control.

### Auth gates

All mutating server functions go through `requireAdminServerFn()`. It returns `{ ...identity, accessToken }` where `accessToken` is the right GitHub token for content commits (OAuth user token or service PAT — the adapter decides). **Never** read `request.cookies` directly outside the auth module.

### Adding env vars

1. Add to the schema in `src/env.ts` (zod, with sensible defaults).
2. Add to `.env.example` with a comment explaining what it does.
3. Document it in the relevant doc (`docs/AUTH.md`, `docs/STORAGE.md`, etc.).
4. Never read `process.env` outside `src/env.ts`. Use `getEnv()`.

### Tests

Mirror the source path. `src/auth/members.ts` → `tests/auth/members.test.ts`. Vitest, not jest. Don't snapshot full HTML — assert on specific behavior.

## Landmines

These are the things that will bite you if you don't know.

### 1. The `useEffect` rule is real

`eslint.config.js` flags `useEffect()` and `React.useEffect()` call sites with `no-restricted-syntax`. If you're reaching for it, stop and ask: can this be derived state, an event handler, a React Query subscription, or `useMountEffect`? In nearly every case the answer is yes. If you genuinely need it, justify it with an `// eslint-disable-next-line no-restricted-syntax` comment.

The codebase currently has **zero** `useEffect` call sites. Keep it that way.

### 2. `import * as React from "react"` is fine

The lint rule used to fire on namespace imports — that was a bug in the old config and is fixed. Namespace imports stay.

### 3. `any` and floating promises are errors

`@typescript-eslint/no-explicit-any: error` and `no-floating-promises: error`. You'll see these fail in CI before `tsc` even tells you why. `await` everything or assign to a `void` cast intentionally.

### 4. `pnpm install --frozen-lockfile` in CI

If you change `pnpm-lock.yaml`, commit it. CI installs frozen.

### 5. `routeTree.gen.ts` is generated

Don't hand-edit. Run `pnpm routegen` (or `pnpm typecheck` which calls it). It's gitignored from prettier; eslint ignores it too.

### 6. Members format

`GITCMS_MEMBERS=identifier:role,identifier:role`. The `:` is allowed inside identifiers (JWT subjects look like `jwt:user_abc`), so the parser splits on the _last_ `:`. Unknown role suffixes are treated as part of the identifier, not a role. See `parseMembersSpec` in `src/auth/members.ts`.

### 7. `admin.accessToken` field name is sacred

Across the auth refactor, every callsite that reads `admin.accessToken` was preserved. Renaming this field will require touching every server function. Don't.

### 8. Services use the GitHub token, not the OAuth user token

When `GITCMS_GITHUB_TOKEN_SOURCE=service`, content commits use a fixed PAT. Auth identity (who's signed in) is still tracked separately — that's why we expose `Co-authored-by` on commits. Never assume "signed-in user" === "commit author."

### 9. The auth adapter is async-init

`buildAuthFromEnv()` (in `src/auth/index.ts`) reads env at startup. If you add a new auth mode, register it there. Don't swap implementations at request time.

### 10. Tests must not hit the real GitHub API

Mock the client. There's no integration test layer in v0.0.x.

## Adding things

### A new auth provider

1. Implement `AuthAdapter` (in `src/auth/your-provider-adapter.ts`).
2. Wire it into `buildAuthFromEnv()` in `src/auth/index.ts`, gated by an env flag.
3. Add env vars to `src/env.ts` and `.env.example`.
4. Add tests in `tests/auth/`.
5. Document in `docs/AUTH.md`.

### A new storage adapter

1. Implement `StorageAdapter` (in `src/storage/your-backend.ts`).
2. Register in `src/storage/index.ts`.
3. Add env vars.
4. Tests in `tests/storage/`.
5. Document in `docs/STORAGE.md`.

### A new server function

1. Add to `src/server/fn/<area>.ts` using `createServerFn`.
2. Validate input with zod (`.inputValidator(...)`).
3. Gate with `requireAdminServerFn()` if it mutates.
4. Use `getRuntimeConfig()` to read collections — never re-import the config.
5. Test in `tests/server/`.

### A new field type for collections

1. Add to `FieldType` in `src/config.ts`.
2. Update the zod schema next to it.
3. Add UI rendering in `src/components/frontmatter-panel.tsx`.
4. Update `docs/COLLECTIONS.md`.

## What NOT to do

- **Don't add a new HTTP framework.** TanStack Start handles both server fns and routes.
- **Don't add a database.** State lives in git. Period.
- **Don't add a queue or background worker.** v0.0.x is request-response only.
- **Don't add an auth library on top of the existing layer** (no NextAuth, no Lucia, no Clerk SDK). The adapter pattern is the integration point.
- **Don't hardcode a vendor in the storage layer.** `StorageAdapter` is the abstraction.
- **Don't expand `gitcms.config.ts` schema unless you have a real user.** It's stable surface.
- **Don't add telemetry/analytics phone-home.** This is a self-hosted CMS. User data stays on user infra.

## When you're stuck

1. Read this file again. Most "I want to add X" answers are above.
2. Read [`docs/AUTH.md`](docs/AUTH.md) for the full auth matrix.
3. Read [`docs/AGENTS-USING-GITCMS.md`](docs/AGENTS-USING-GITCMS.md) if your question is "how does an agent author content."
4. Open an issue with the question. The maintainers prefer questions to surprise PRs.
